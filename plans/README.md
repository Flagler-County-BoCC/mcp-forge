## Product-pivot plan set (007–010) — added 2026-06-18

Plans 007–010 implement the owner's pivot: mcp-forge should not only *rewrite*
existing projects but *create* hardened, standardized MCP servers from an API
spec, with field-selection, target-aware command generation, and cross-OS
installers. The key architectural insight (see plan 007): the `AUDIT_MANIFEST` is
the single contract every downstream step reads, so "Create mode" is just an
alternate way to *produce* that manifest — the rest of the pipeline is reused
unchanged. Decisions locked in this session: OpenAPI-or-BUILD_SPEC input;
client-side field projection always; raw commands auto + business commands from an
explicit list only.

**Recommended execution order (by leverage, lowest-risk first):**

1. **010** (S, LOW) — propagate the already-proven atomic-write fix into the generator.
2. **009** (M) — fix the confirmed meta-command bug; two-tier command generation.
3. **008** (M) — field-selection on every tool.
4. **007** (L) — Create mode + determinism. Biggest; do last so the smaller wins land first.

**Conflict warning (same lesson as the 001–006 run):** plans **008, 009, 010 all
edit `entrypoints/mcp-server.md`**. Execute them on ONE chained branch (or
sequentially), not as three independent worktrees off HEAD, or they will conflict
on that file. Plan **007** is largely disjoint (it adds `shared/00-create.md`,
edits `MASTER.md`, and touches `src/tools/steps/*` + tests — none of which 008/009/010
touch) and can run on its own branch in parallel.

## Verification reality for 007–010

These plans edit the **prompt library** (the instructions that generate other
projects), not mcp-forge's runtime — so forge's own test count is unchanged by
008/009/010 (007 adds 2 tests for its new served tool). There is no automated test
that *generated* code is correct (that needs an end-to-end generation run). The
machine-checkable gates used instead: grep presence/absence checks, standalone
`tsc --strict` typechecks of embedded code examples (008, 010), and the existing
`prompt-files.test.ts` + `format:check` (prompt dir is prettier-ignored, so prompt
edits never break format). Generated-code *determinism* is validated by reviewing
the explicit derivation rules, not by a unit test.

## Findings considered and rejected

- **Path traversal in `loadPrompt`** (`src/lib/prompt-loader.ts`): not a
  vulnerability — every path passed to it comes from the hard-coded
  `STEP_DEFINITIONS`/`ENTRYPOINT_MAP` internal maps; no tool input reaches
  `path.resolve`. Not worth a guard.
- **Redundant `ProjectTypeSchema.parse()` in tool handlers** (`steps.tool.ts`):
  the MCP SDK already validates inputs via the zod `.shape`, making the re-parse
  belt-and-suspenders. Left in place — removing it changes validation behavior
  for marginal gain and isn't worth the risk. (Noted in plan 005's out-of-scope.)
- **Graceful `server.close()` on SIGTERM** (`src/stdio.ts`): for a stdio server
  with no open connections or in-flight state, `process.exit(0)` is adequate.
  Not worth the ceremony.
